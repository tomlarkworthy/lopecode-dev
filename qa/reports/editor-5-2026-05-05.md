## Summary

**Three independent bugs found across two QA sub-passes (initial + resumed-after-browser-recovery).** The headline: ▶️ on a `viewof` *or* a `mutable` cell freezes the page main thread for >30 s, times out CDP, and disconnects the channel — silent hang, no thrown error. Mutable additionally **shatters the user's single-line declaration into three separately-edited cells** in the UI. Imports use the same `del+new+new` history shape as viewof/mutable but compile cleanly without hanging. The `del+new+new` shape itself violates the "Pid stable for `op:"upd"`" invariant in the per-notebook guidance whenever a cell's *kind* changes — across viewof, mutable, and import. Clean-state pid uniqueness for two consecutive ➕ still holds (#144 not regressed). Cell-move, syntax errors, and type-conversion sequences are still owed.

## Issues found

### 0. Mutable cells: ▶️ hangs the page AND the UI fragments into three separate cells — severity: high — criterion: #4 / #5
- **What:** Two distinct symptoms from a single `mutable qa_counter = 0` cell:
  - **(a) Same hang as viewof.** Compiling a `mutable X = ...` definition through ▶️ freezes the page main thread for >30 s; `qa_screenshot` and `eval_code` both time out; channel emits `disconnected`. Identical signature to the viewof hang (issue #1) — confirms the freeze isn't viewof-specific, it's anything that materialises a *paired-variable* shape (viewof: 2 paired; mutable: 3 paired). Imports (issue #4 below) also produce paired variables but **don't** hang, so the trigger isn't strictly "paired" — there's something specific to viewof+mutable's eager evaluation path.
  - **(b) UI shatters the single declaration into three editable cells.** The user typed one line, `mutable qa_counter = 0`. After ▶️ the editor pane shows **three separately-listed cells**: `initial qa_counter`, `mutable qa_counter`, and `qa_counter` — each with its own ⬆⬇🗑▶️📋 hotbar. The user reported seeing this directly: *"I saw a UI element called 'initial mutable = 4', meaning the single cell got shattered into its underlying representation of three low level variables."* Editing any of the three (other than `initial qa_counter`, which holds the user's authored expression) desyncs the mutable invariant — `mutable qa_counter` and `qa_counter` carry the runtime stitching `(M, _) => new M(_)` and `_ => _.generator` and aren't supposed to be hand-edited. The implementation-detail variables shouldn't be exposed as authorable surface.
- **Where:** `@tomlarkworthy/editor-5` recompile path (the hang), and the `auto_attach` / cell-list rendering path (the fragmentation). The fragmentation is almost certainly because `auto_attach` enumerates `runtime._variables` and attaches a hotbar to each — without recognising that `initial X` / `mutable X` / `X` came from one source declaration and should share one editor.
- **Evidence:**
  - History recorded the three `op:"new"` entries as expected — these are correct at the runtime level:
    ```js
    {op:"new", pid:"_49yu0tk", _name:"initial qa_counter", _definition:"function _qa_counter() {return (0);}"}
    {op:"new", pid:"_95fb0vj", _name:"mutable qa_counter", _inputs:["Mutable","initial qa_counter"], _definition:"(M, _) => new M(_)"}
    {op:"new", pid:"_bukdzc4", _name:"qa_counter",         _inputs:["mutable qa_counter"],            _definition:"_ => _.generator"}
    {op:"del", pid:"_4qxspst", _name:null}
    ```
    The history is correct; the UI rendering is what's wrong. The editor should treat the three-pid bundle as one editable cell.
  - User-observed UI fragmentation: three hotbars, one per variable. Confirmed in-session.
  - Page hang: `qa_screenshot` and `eval_code` both timed out at 30 s; channel emitted `disconnected`. No console errors, no failed network requests. Same silent-hang signature as issue #1.
- **Hypothesis:**
  - For (a) — the hang — same as issue #1: the synchronous walk over editor-5's ~1010 scoped names triggered by every new variable, compounded when one declaration creates three. The fix is presumably to coalesce auto_attach's reattach pass (debounce, or batch by source declaration).
  - For (b) — the UI shatter — `auto_attach` should group by source-declaration provenance: variables that share an authoring origin (`mutable qa_counter` declaration → `initial qa_counter` + `mutable qa_counter` + `qa_counter`) should expose a single editor, not three. The provenance is reconstructible from `_inputs` (each derived variable points back to the `initial X` pid), but the editor doesn't currently use it.
- **Why it matters:** Mutables and viewofs are two of the three idioms a user writing "real" cells regularly reaches for (the third being imports). One of them deletes the user's editing context for 30+ seconds; the other does the same AND also reframes the user's mental model from "one cell that mutates" to "three cells, two of which I shouldn't touch but the editor lets me click into them anyway." The "Edit any cell in the runtime" feature claim breaks down at exactly the point where editing matters most.

### 1. ▶️ on a `viewof` cell freezes the page main thread — severity: high — criterion: #4 / #10
- **What:** Compiling a `viewof X = ...` definition through the editor toolbar's Run button hangs the page's main thread long enough that:
  - `qa_screenshot` (CDP framebuffer capture) times out at 30 s.
  - `eval_code('1+1')` over the channel times out.
  - The pairing channel emits `disconnected` (channel transport heartbeat lapses).
  - Subsequent `qa_open_notebook` calls into the same notebook also time out at `goto`, suggesting the runtime crashed badly enough that even a fresh navigation cannot complete promptly. (May or may not be the same root cause; not isolated yet.)
- **Where:** `@tomlarkworthy/editor-5` Run handler (▶️ button) → cell recompile pathway. Almost certainly the codepath that materialises `viewof X` into the two paired runtime variables (`viewof X` for the DOM, `X` for `Generators.input(_)`).
- **Reproduced twice in this pass:**
  - **Repro A — fresh viewof cell:** ➕ → click "edit" on new empty cell → type `viewof slider = Inputs.range([0, 100], {value: 42, step: 1, label: "QA slider"})` → click ▶️. History fires `new viewof slider _t11wxcb`, `new slider _m34clcc`, `del _hrq3606` simultaneously, then the page locks.
  - **Repro B — convert existing cell:** ➕ → edit to `qa_simple = 42` → ▶️ (works fine, value resolves to `42`) → re-edit to `viewof qa_slider = Inputs.range([0, 10], {value: 3, label: "qa"})` → ▶️. History fires `new viewof qa_slider _86gobsm`, `new qa_slider _g7tnwcm`, `del _wl9otsp` simultaneously, then the page locks.
- **Evidence:**
  - `history` events captured in both repros (excerpts above and full payload in channel transcript).
  - `qa_console_logs` returned `errors: []`, `failed_requests: []` — i.e. **no thrown error**, no rejected promise, no failed network request. The hang is silent: the runtime isn't reporting failure, it's just busy on the main thread.
  - A simple `qa_simple = 42` cell ▶️'d from the same shell in the same session in <100 ms with a clean `op:"upd"` and no hang, so the symptom is specific to `viewof`, not to ▶️ in general or the loaded module weight.
- **Hypothesis:** Compiling a viewof in editor-5 creates the *paired* runtime variable (`Generators.input(_)`) eagerly. If that path interacts with `auto_attach`'s observer cascade (every newly-created variable triggers a re-render of every dynamic editor in the visualizer), the second cell's creation re-walks the editor-5 dependency graph at quadratic cost. Editor-5 reports ~1010 scoped names; even one synchronous walk is hot. A prior `cellEditor(title_variable)` demo only opens one editor on a known cell, so this codepath was not exercised in the earlier passes. Worth profiling the recompile in the inspector — the time budget is almost certainly inside one synchronous event handler.

### 2. Type-changing edits emit `del + new + new`, not `upd` — severity: medium — criterion: #4 (history invariants) / docs
- **What:** Editing a cell *in place* such that its declaration *kind* changes (regular → viewof) is recorded in `history` as `del` of the old pid plus two `new` entries for the new viewof + extracted-value pair, instead of `upd` on the original pid. Per `qa/per-notebook/editor-5.md` "Mutations must correlate to history" §3, the documented invariant is *"Pid stability for `op:"upd"` — editing the same cell twice must produce two `op:"upd"` entries with the same `pid`."* That invariant is **violated by any kind-change edit**.
- **Where:** `@tomlarkworthy/editor-5` recompile pathway → `@tomlarkworthy/local-change-history` event emitter.
- **Evidence (from Repro B above):**
  ```js
  // Two ops earlier: "upd" on _wl9otsp with _name "qa_simple", _definition "function _qa_simple() {return (42);}"
  // After re-edit + ▶️:
  { op:"new", pid:"_86gobsm", _name:"viewof qa_slider", _inputs:["Inputs"], _definition:"function _qa_slider(Inputs) {return (Inputs.range(…))}" }
  { op:"new", pid:"_g7tnwcm", _name:"qa_slider",        _inputs:["Generators","viewof qa_slider"], _definition:"(G, _) => G.input(_)" }
  { op:"del", pid:"_wl9otsp", _name:"qa_simple" }
  ```
- **Hypothesis:** The recompile path probably tears down the original Variable and rebuilds, emitting `delete` then `define` against the runtime, and the local-change-history listener mirrors those calls 1:1 instead of collapsing them into an update. For viewof in particular this is unavoidable (the new shape *has* two backing variables), but emitting `del+new` for plain edits when only the cell name changed is ugly because the orphaned pid lingers in any external reference (cached UI state, the editor's own `editedCell`, persistence tooling that keys by pid). Either:
  - (a) the editor should detect "kind change" and explicitly emit a typed `convert` op so consumers can stitch old→new, or
  - (b) the per-notebook guidance should be relaxed to say "pid stability for upd applies only when the cell *kind* is unchanged."
- **Why it matters:** Anything that snapshots history and expects deletes to mean "user removed the cell" will misinterpret a plain edit as a deletion. That's a common assumption for diff-and-undo tooling.

### 4. Imports: same `del + new + new` history shape, but no hang — severity: low — criterion: #4 (history) / #11 (cross-module noise)
- **What:** Editing an empty cell to `import {Range} from "@tomlarkworthy/view"` and clicking ▶️ produces:
  ```js
  {op:"new", pid:"_6pmjrm1", _name:"module @tomlarkworthy/view", …}
  {op:"new", pid:"_luzw1rb", _name:"Range",                       _inputs:["module @tomlarkworthy/view","@variable"], _definition:'(_, v) => v.import("Range", _)'}
  {op:"del", pid:"_i4c9y81", _name:null}
  {op:"new", pid:"_37amgw",  _name:"Range",                       module:"@tomlarkworthy/view", _definition:"() => x"}
  ```
  Three observations:
  1. Same `del+new+new` shape as viewof + mutable. The "Pid stable for `op:"upd"`" invariant is violated whenever a cell *kind* changes — confirmed not specific to viewof. Already documented in issue #2 above; this strengthens the documentation evidence.
  2. **Page does NOT hang.** `Range = ƒ()` was readable via `get_variable` immediately after, screenshot was instant, channel stayed connected. Imports produce paired runtime entries (`module @tomlarkworthy/view` + `Range`) but evidently exit the editor's recompile path before the auto_attach cascade can turn quadratic — likely because the import resolution is asynchronous (`async () => runtime.module(...)`), which yields the main thread.
  3. **Cross-module history pollution.** `op:"new"` was recorded for `Range` in `@tomlarkworthy/view` (a module the user never directly edited). `local-change-history` watches every module in the runtime and faithfully reports *any* new variable creation, even ones that happen as a side effect of an import in another module. A diff-and-undo tool keying off `module` will see edits the user didn't actually make. Probably not actionable on its own, but worth noting.
- **Where:** `@tomlarkworthy/editor-5` import-resolution path → `@tomlarkworthy/local-change-history` listener.
- **Hypothesis:** The async-vs-sync split between import and viewof+mutable in the recompile path is the most likely explanation for why two of the three trigger the hang and one doesn't. Forcing viewof/mutable through an `await Promise.resolve()` between paired-variable creations would probably fix the hang without changing observable semantics — though it'd be better to fix the underlying auto_attach quadratic.
