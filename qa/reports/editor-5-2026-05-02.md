## Issues found

### 1. Cmd/Ctrl+Enter does not recompile the cell — severity: medium — criterion: #4
- **What:** Standard "execute current cell" shortcut is unbound. Typed " QA" into the title editor, pressed `Meta+Enter` — editor source updated, but runtime did **not** recompile (H1 unchanged). After then clicking the toolbar Run (▶️) button, the runtime *did* recompile (H1 caught up). This was reproduced a second time with a single-character "!" edit: `Meta+Enter` → no `history` `op:"upd"` event; subsequent click on (159, 703) → immediate `op:"upd"` event.
- **Where:** `@tomlarkworthy/editor-5` toolbar / CodeMirror keymap.
- **Evidence:** Channel `history` watcher fired only on toolbar clicks, never on `Meta+Enter`; H1 textContent confirmed unchanged after the keypress.
- **Hypothesis:** The CodeMirror 6 keymap doesn't bind `Mod-Enter` to the `compile_and_update` runtime command. Adding a simple keybinding (`{key: "Mod-Enter", run: () => command_processor("run")}`-shaped) would close the gap. This was previously listed as an "open question" in per-notebook guidance — now confirmed missing.

### 2. Toolbar buttons have no tooltips or `aria-label` — severity: medium — criterion: #4 / a11y
- **What:** All seven editor-toolbar buttons are bare emoji glyphs in the DOM (`<button>⬆</button>`, `<button>⬇</button>`, `<button>🗑️</button>`, `<button>➕</button>`, `<button>▶️</button>`, the two-page save icon, `<button>📋</button>`). None has `title=""` or `aria-label=""`. Sighted users have to guess; screen readers read the emoji name (e.g. "up arrow"), not the function ("move cell up"). The arrows in particular have no visible feedback at boundaries (verified prior pass), so they're doubly opaque.
- **Where:** `@tomlarkworthy/editor-5` toolbar HTML emission.
- **Evidence:** `outerHTML` of buttons via `eval_code` showed empty `title` and null `aria-label`.
- **Hypothesis:** Adding `title="Run cell"` etc. is a 7-line change and would resolve both the discoverability and a11y gaps.

### 3. Failed network request: `notebookwebhook.mov` (×2) — severity: low — criterion: #4
- **What:** Two `GET https://storage.googleapis.com/publicartifacts/blogimages/notebookwebhook.mov` aborted with `net::ERR_ABORTED` shortly after page load. Identical to prior pass — not regressed, not fixed.
- **Where:** Asset reference somewhere in the editor-5 markdown (not visible in the default layout).
- **Evidence:** `failed_requests[2,3]` in `qa_console_logs` JSON (timestamps 1777757343592 and 1777757345002).
- **Hypothesis:** Asset deleted from GCS; reference should be removed or replaced. Likely affects more than this notebook — worth a corpus-wide grep.

### 4. `blob:null/...` aborted GETs (×2) on load — severity: low — criterion: #4
- **What:** Two `GET blob:null/<uuid>` requests fail with `net::ERR_ABORTED` during boot. Identical to prior pass.
- **Evidence:** `failed_requests[0,1]`.
- **Hypothesis:** Bootloader cleanup race — blob URL revoked while a fetch is still in flight. Cosmetic; creates noise in DevTools.

### 5. CSS `@import` warnings (×3) — severity: low — criterion: #4
- **What:** Three identical warnings: `@import rules are not allowed here. See https://github.com/WICG/construct-stylesheets/issues/119#issuecomment-588352418.` Identical to prior pass.
- **Evidence:** `console[].type=warning` × 3.
- **Hypothesis:** One of the upstream `notebook-kit/styles/*.css` files uses `@import`, and the constructed-stylesheet path silently drops it. Either rewrite the upstream sheet to inline its imports, or fall back to `<link>` for sheets that need them.
