## robocoop-4 findings (backlog B1–B3)

- **B3 (v5, functional):** an agent run **hard-freezes the notebook**. After llama wrote the file, the pairing channel went fully unresponsive for 6+ min; even `eval(1+1)` times out → a synchronous main-thread wedge, not slow serialization. `maxStepsPerTurn:20` did not recover it. Report content unrecoverable (virtual fs, file-sync was off). Needs root-causing in `robocoop-4-core.js` (tool loop / render path).
- **B1 (v4, functional):** **agent errors fail silently** in the chat UI. Six 402-triggering submits each cleared the input and re-enabled Send but rendered **no error bubble** (`#2d0f0f`) and no assistant reply. `submit` catches into `errors[]` and `render()` has an error-bubble path, but it isn't surfacing. Success turns render fine → defect is the error-render path. A user with an out-of-credit / rate-limited key sees their message vanish.
- **B2 (v4, performance):** engine **omits `max_tokens`**, so the provider reserves the model's full 32k output budget → the opus 402 above. `core.js chat()` only sends `max_tokens` when non-null; the engine never sets it. A sane default (~8k) would fix budget-limited keys and cut cost.

## lopeteam SKILL findings (feed `/lopeteam-reflect`)

1. **Two-browser gap.** Setup uses `open_url` (system browser → introspection works) but the aesthetic critic's `qa_screenshot`/`qa_*` need `qa_open_notebook` (Playwright). Different browsers. Fix: open via `qa_open_notebook` when visual QA is in scope.
2. **Parallel-fan-out is overstated.** Functional/aesthetic/performance critics all drive the single live browser; only disk-static analysis (code-quality, upstream) is truly parallel-safe. The role split is right; the "spawn 5 in parallel" claim must be corrected to "fan out disk-static reads in parallel; serialize channel-driving reads."
3. **Functional probe is heavy + stateful + serialized.** "Assess by running the agent" is a long, channel-owning, paid operation — it belongs in the serialized apply/verify lane, not the read-only parallel lane. And **a running agent can wedge the channel** (see B3), so the orchestrator needs a watchdog/timeout + the ability to recover (reload) rather than assuming channel introspection stays live during a run.
4. **Need a pre-run baseline guard.** The target file was already `M` (uncommitted) at start; a real run must branch/worktree off a clean baseline before applying.
5. **Channel reads can hard-timeout.** `get_variable`/`eval_code` block for 30s with no partial result when the page is busy/frozen; the loop must treat repeated timeouts as a stall signal (feeds dry-counter) and surface "page may need reload".

## Recommended next steps

- For the real opus-4 assessment: top up the OpenRouter key **or** lower the engine `max_tokens` default (fixes B2 and unblocks opus within budget).
- Root-cause B3 (the freeze) before any further functional probing — it's the highest-value fix and currently makes the notebook unusable for real tasks.
- Reload the frozen tab before the next session.
