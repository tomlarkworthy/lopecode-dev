## editable-md — durable notes

### Known issues / quirks

- **Smart-quote corruption in `${…}` interpolations (high severity).** ProseMirror's `exampleSetup` enables `smartQuotes`, which rewrites straight `'`/`"` to U+2018–U+201D as the user types. Inside an interpolation block this produces uncompilable JS; `compile()` throws `SyntaxError: Unexpected character '“'`, the cell is replaced with a `throw …` stub, and the stub is persisted to `local-change-history` → survives reload. See `qa/reports/editable-md-2026-05-19.md` issue #1. Don't be surprised by `title = SyntaxError: Unexpected character '“'` on a "fresh" reload — it's the sticky-corruption signature, not a fresh failure. Recovery requires hand-editing the JS cell (or clearing the local git branch). Re-test the fix by typing `${"x"}` inside any md cell, SHIFT+ENTER, and confirming the cell still renders.
- **Cell-corruption is persistent.** Any compile failure on commit is captured by local-change-history with `provenance: {source: "git", repo: "/@user/module"}` and re-applied on reload. The renderer has no in-prose undo. When you encounter a broken cell here, check the channel `history` events first — the `provenance` field tells you whether the corruption came from the current session or from local git.
- **`isInteractiveTarget` may not catch all link clicks.** Clicking an `[href]` anchor inside rendered markdown opened the editor on at least one occasion (Known Issues bullet's `#test_defaultMarkdownParser_preserves_escapes`). The selector includes `a[href]` but the click outcome was edit-mode-open, not navigation. Worth a deeper repro if/when someone changes the listener; not user-blocking.
- **Drag-to-select inside prose mounts the editor and wipes the highlight (high severity).** The `md` cell attaches a plain `click` listener with no selection-aware gate. After a drag-select, the browser fires `click` at mouseup, the listener proceeds, `dom.innerHTML = ""` runs, and ProseMirror remounts — losing the user's selection. Reproducible with a 3-char range and trivially with longer ranges (verified 2026-05-19, `editable-md-2026-05-19b.md` issue #1). The user has no way to highlight text inside any rendered `md` cell. Probe before re-testing fixes: install a Range via `getSelection().addRange()`, dispatch `click`, assert `.ProseMirror` did NOT mount and `getSelection().toString()` is preserved. Don't be fooled by "the cell rendered fine" — the surface looks correct until you try to highlight.
- **First click of a double-click also mounts the editor (medium severity).** Browser fires two `click` events for a dblclick; the first has `hasSelection:false` (word-select happens between them at mousedown), so a selection-empty guard alone does NOT fix the dblclick case. If the planned fix is selection-only, also gate on `event.detail === 1` or wire a `dblclick` pre-empt — otherwise users who instinctively double-click to word-select still land in the editor (verified 2026-05-19, `editable-md-2026-05-19b.md` issue #2).

### Behavioral facts worth knowing

- **Two commit triggers, only one documented.** Prose mentions SHIFT+ENTER; focusout (clicking outside the cell) also commits via `lostFocus()`. Test both when verifying commit logic — they share `saveAndRender()`.
- **`dom.markdown` getter returns a Promise.** Resolves to the *current* serialized markdown (it re-runs `decompile` + `mdCellSourceToMarkdown` each call, not cached). Useful for QA snapshotting; don't assume it's a property.
- **Title cell holds the slider interpolation.** The "value is bound to markdown document" Inputs.range (`viewof control`) feeds `${control}` in the title cell — that's what produces "the value of the slider is: 0.5" in the rendered prose. Move the slider → title re-renders → `dom.markdown` updates. Useful as a sanity check that reactivity is alive.

### QA setup specifics

- Hash from bootconf: `#view=S100(@tomlarkworthy/editable-md,@tomlarkworthy/module-selection)` — keep it; the module-selection tab is a sibling, not a child.
- The page boots with `Bootloader: loading @tomlarkworthy/lopepage` + `… loading @tomlarkworthy/editable-md`; `keepalive: dynamic observe …` chatter is expected (~50 lines).
- `mcp__lopecode__eval_code` returns `undefined` even on success — pipe results via `console.log(...)` and read with `qa_console_logs` instead of relying on the return value.
- Multiple variables share the name `title` across modules (bootloader's H1, editable-md's, runtime-sdk's docstrings, etc.). When introspecting, filter by `_module` or by `_inputs`/`_definition` shape, not by `_name` alone.
- Five `test_*` cells exist. Only `test_mdCellSourceToMarkdown` resolves to a non-`undefined` value (it's an expression cell that returns the converted markdown string). The other four are `expect(...)`-based — pass = `hasError: false, _value: undefined`. Don't read absence of value as a pending test.
